# Supabase Migration Failure Recovery

## Incident Summary
The initial attempt to deploy the Supabase Prisma migration failed due to a schema engine error: `error encoding message to server: string contains embedded null` and `syntax error at or near "-"`. 

This occurred because the local migration SQL file was originally generated by redirecting standard output using PowerShell's default behavior, which encoded the file in UTF-16LE. The UTF-16LE format introduces null bytes for standard ASCII characters, causing the Prisma Migration engine to choke when submitting queries to Supabase. Later attempts to fix it inadvertently included terminal log outputs from `dotenvx` at the beginning of the SQL script.

## Migration Tracking
- **Corrupted Migration Name**: `20260716113020_initial_supabase_schema`
- **Canonical Repaired Migration Name**: `20260716113020_initial_supabase_schema` (The original folder was preserved as canonical, but its contents were overwritten with the exact UTF-8 output directly generated from `prisma migrate diff`).

## Partial State & Cleanup
- **Partial Objects Discovered**: Due to failing on the very first bytes of the file, no application tables or enums were partially created. Only the `_prisma_migrations` table existed.
- **Exact Cleanup Objects**: Although no objects existed, a targeted, foreign-key-safe `DROP TABLE IF EXISTS` and `DROP TYPE IF EXISTS` transaction for the 19 tables and 8 enums was executed against the `public` schema. It verified no data existed and strictly avoided dropping any Supabase-managed objects or schema configurations.

## Resolution
- **Migrate Resolve Result**: The failed migration attempts were successfully marked as rolled back using `npx prisma migrate resolve --rolled-back 20260716113020_initial_supabase_schema`.
- **Migrate Deploy Result**: After repairing the file to clean UTF-8 with no null bytes or injected logs, `npx prisma migrate deploy` successfully applied all schema changes.
- **Final Migration Status**: `Database schema is up to date!`. The database now correctly holds 19 tables, 8 enums, and the associated foreign keys/indexes.

## Prevention
To prevent this in future migrations, a validation script `scripts/validate-migrations.ts` has been added. This script actively checks the `prisma/migrations` folder for any `.sql` file containing null bytes and causes the process to fail before deployment if corrupted SQL is detected. In addition, future schema generation scripts now pipe directly through Node.js filesystem modules rather than relying on host shell output redirection.
